## 1. 비전 & 핵심 개념

### 1.1 비전

재귀적 멀티 에이전트 시스템. 회사 조직 구조처럼 동작하는 에이전트들이 계층적으로 협업하여 복잡한 작업을 수행한다.

### 1.2 핵심 개념

**Hollon(홀론)**: 전체이자 부분인 자율적 에이전트 단위

- Arthur Koestler의 홀라키(Holarchy) 개념에서 영감
- 각 홀론은 Brain Provider(Claude Code 등)를 통해 실행됨

**Organization(조직)**: 홀론들의 최상위 컨테이너, 리소스/비용 제한 관리

## 3. 핵심 엔티티

| 엔티티           | 설명                                        | 회사 비유   |
| ---------------- | ------------------------------------------- | ----------- |
| **Organization** | 최상위 컨테이너, 비용/리소스 제한           | 회사        |
| **Role**         | 역할 템플릿 (시스템 프롬프트, 권한, 자율성) | 직무 기술서 |
| **Team**         | 홀론들의 그룹, 계층 구조 지원               | 부서/팀     |
| **Hollon**       | Role 기반 에이전트 인스턴스                 | 직원        |
| **Goal**         | OKR 구조 목표 (인간 또는 자율 생성)         | 분기 OKR    |
| **Project**      | Task 그룹화 (선택적)                        | 프로젝트    |
| **Task**         | 재귀적 작업 단위, Project 없이도 생성 가능  | Linear 이슈 |
| **Milestone**    | Task의 선택적 그룹핑 (폴더 역할)            | 폴더/라벨   |
| **Cycle**        | 시간 기반 작업 단위, 예산 추적              | 스프린트    |
| **Channel**      | 그룹 커뮤니케이션 공간                      | Slack 채널  |
| **Document**     | Markdown 문서 + Long-term Memory            | Wiki/Notion |
| **Message**      | 홀론 간 1:1 커뮤니케이션                    | DM          |

### 4.2 프롬프트 합성 과정

홀론이 Task를 실행할 때, 6개 계층의 정보가 자동으로 합성됩니다:

```
┌─────────────────────────────────────────────────────────────────┐
│                    최종 프롬프트 (Brain에 전달)                    │
├─────────────────────────────────────────────────────────────────┤
│  [Layer 1] Organization: "우리는 사용자 중심 제품을 만든다..."    │
│  [Layer 2] Team: "Engineering팀은 코드 리뷰 필수, TDD 준수..."    │
│  [Layer 3] Role: "당신은 백엔드 엔지니어로서 API 설계 담당..."     │
│  [Layer 4] Hollon: "TypeScript와 NestJS에 특화된 시니어..."       │
│  [Layer 5] Memory: "[API 설계 가이드], [이전 인증 구현 기록]..."   │
│  [Layer 6] Task: "Task #42: JWT 기반 인증 미들웨어 구현..."       │
└─────────────────────────────────────────────────────────────────┘
```

**Memory 주입 과정:**

1. Task 키워드 추출 ("JWT", "인증", "미들웨어")
2. 관련 Document 검색 (scope 매칭 + keywords 매칭)
3. importance 점수순 정렬
4. 컨텍스트 한도 내 상위 문서 선택

### 4.3 태스크 처리 분기

```
태스크 수신
    │
    ├─ 복잡도 분석 (LLM 판단)
    │   ├─ 예상 토큰
    │   ├─ 필요 도메인
    │   └─ 분할 가능성
    │
    └─▶ 분기 결정
        │
        ├─ [단순] → 직접 수행 → 결과 Document 저장
        │
        ├─ [복잡] → 서브태스크 생성 → Task Pool에 추가
        │           (부모 태스크는 PENDING 상태로 대기)
        │           │
        │           └─▶ 서브태스크 모두 완료 시 (Phase 3.10)
        │                 │
        │                 └─ 부모 태스크 → READY_FOR_REVIEW
        │                       │
        │                       └─▶ 리뷰 사이클 (우선순위 0)
        │                             │
        │                             ├─ 부모 홀론이 Priority 0으로 풀
        │                             ├─ IN_REVIEW 상태로 전환
        │                             ├─ LLM이 서브태스크 결과 검토
        │                             │
        │                             └─▶ 4가지 결정
        │                                   │
        │                                   ├─ [complete] → COMPLETED
        │                                   │   └─ 임시 홀론 정리
        │                                   │
        │                                   ├─ [rework] → 특정 서브태스크 READY로 재설정
        │                                   │   └─ 재작업 지시사항 포함
        │                                   │
        │                                   ├─ [add_tasks] → 후속 서브태스크 추가
        │                                   │   └─ 부모 태스크는 계속 PENDING
        │                                   │
        │                                   └─ [redirect] → 일부 서브태스크 취소
        │                                       └─ 부모 태스크 방향 전환
        │
        └─ [불확실] → Spike 생성 (Time-boxed 실험)
                      → 결과에 따라 본 작업 결정
```

**리뷰 사이클 특징 (Phase 3.10):**

- 무한 루프 방지: reviewCount 최대 3회 제한
- 우선순위: Priority 0 (직접 할당 태스크보다 높음)
- 컨텍스트 보존: 부모 홀론이 자신의 컨텍스트에서 리뷰
- 임시 홀론 정리: complete 결정 시 생성한 임시 홀론 자동 삭제

```

## 5. 핵심 원칙

### 5.1 Single Context 원칙

홀론은 **하나의 컨텍스트에서 하나의 태스크**를 완료하고, 다음 태스크를 가져옵니다.

**이점:**

- 컨텍스트 유실 문제 해결 (매 태스크가 독립적)
- 명확한 작업 단위와 측정 가능한 성과
- 실패 시 영향 범위 최소화
- 병렬 처리 용이

### 5.2 동시성 모델 (Concurrency)

홀론 간 협업은 **병렬성(Parallelism)이 아닌 동시성(Concurrency)**으로 동작합니다.

```

인간 회의: A 발언 → B 듣고 생각 → B 발언 → A 듣고 생각 → ...
홀론 협업: A 작업 → DB 저장 → B 읽음 → B 작업 → DB 저장 → ...

차이: 지연 시간(latency)뿐, 본질은 동일한 순차적 컨텍스트 스위칭

```

LLM은 동시에 "생각"할 수 없으므로, Document-Memory 기반 비동기 협업이 실시간 논의와 **개념적으로 동일**합니다.

### 5.3 6계층 프롬프트 합성

| 계층    | 소스                 | 역할                            |
| ------- | -------------------- | ------------------------------- |
| Layer 1 | Organization Context | 회사 비전, 핵심 가치, 전사 규칙 |
| Layer 2 | Team Context         | 팀 목표, 협업 규칙              |
| Layer 3 | Role Prompt          | 역할 정의, 전문 지식, 권한      |
| Layer 4 | Hollon Custom        | 개별 홀론 성격, 특화 영역       |
| Layer 5 | Long-term Memory     | 관련 Document (자동 선택)       |
| Layer 6 | Task Context         | 현재 태스크 정보                |

### 5.4 Brain Provider 아키텍처

홀론의 "두뇌"는 다양한 AI 백엔드를 지원하도록 추상화됩니다.

| Provider               | 카테고리 | 코드실행 | 파일시스템 |
| ---------------------- | -------- | -------- | ---------- |
| **claude-code** (기본) | CLI      | ✅       | ✅         |
| gemini-cli             | CLI      | ✅       | ✅         |
| claude-api             | API      | ❌       | ❌         |
| gemini-api             | API      | ❌       | ❌         |

**선택 우선순위**: `Hollon 설정 > Role 설정 > Organization 기본값`

### 5.5 Document-Memory 통합

Document 시스템이 Long-term Memory 역할을 겸합니다.

| 속성               | 설명                                   |
| ------------------ | -------------------------------------- |
| **scope**          | organization / team / project / hollon |
| **doc_type**       | spec, adr, guide, memory, postmortem   |
| **keywords**       | 검색/매칭용 키워드 배열                |
| **importance**     | 1-10 중요도 (주입 우선순위)            |
| **auto_generated** | 자동 생성 여부                         |

> **온보딩**: 새로운 홀론은 Document-Memory 시스템을 통해 프로젝트 컨텍스트, 코드 컨벤션, 아키텍처 결정 등을 즉시 학습하여 별도 온보딩 없이 업무 투입 가능

### 5.6 하이브리드 문서 검색

문서 검색은 **Vector RAG + Graph RAG** 하이브리드 방식을 사용합니다.

| 방식           | 용도                  | 예시                                             |
| -------------- | --------------------- | ------------------------------------------------ |
| **Vector RAG** | 의미적 유사 문서 검색 | "인증 관련 문서" → OAuth, JWT, Session 문서 반환 |
| **Graph RAG**  | 관계 기반 탐색        | "이 ADR의 관련 문서" → 참조/의존 문서 체인 반환  |
| **Hybrid**     | 통합 검색             | 두 결과를 RRF로 병합                             |

**문서 관계 유형**: `references`, `depends_on`, `supersedes`, `related_to`

## 6. 자율 운영 정책

### 6.1 운영 철학: "사람은 이사회"

인간은 전략적 결정과 예외 상황에만 개입합니다. 일상적인 운영은 홀론들이 자율적으로 수행합니다.

### 6.2 인간-홀론 역할 분담

| 영역          | 🧑 인간 (승인 필요)                | 🤖 홀론 (자율)                    |
| ------------- | ---------------------------------- | --------------------------------- |
| **조직 관리** | 영구 홀론 생성/삭제                | 임시 홀론 생성/종료               |
| **전략**      | 전략 방향 변경, Goal 설정          | 태스크 할당/처리, 서브태스크 생성 |
| **리소스**    | 예산 초과 승인                     | 예산 범위 내 작업                 |
| **외부 영향** | 외부 이해관계자 영향, 법적/규정    | 내부 시스템만 영향                |
| **되돌림**    | 되돌릴 수 없는 결정                | 되돌릴 수 있는 결정               |
| **피벗 대응** | 피벗 결정, Goal 폐기/수정          | 영향 분석, 작업 중단, 자산 분류   |
| **파일/API**  | 허용 외 파일 접근, 미승인 외부 API | 허용된 경로/API 접근              |

### 6.3 에스컬레이션 계층

```

Level 1: 자기 해결 (재시도, 대안 탐색)
↓ 실패 시
Level 2: 팀 내 협업 (같은 팀 홀론에게 도움 요청)
↓ 해결 불가 시
Level 3: 팀 리더 판단 (우선순위 조정, 리소스 재배치)
↓ 권한 밖 시
Level 4: 상위 팀/조직 레벨
↓ 중대 사안 시
Level 5: 인간 개입 요청

```

| ---------------- | ---------------------- | ---------------------- |
| 태스크 완료 검증 | 결과물 존재, 포맷 준수 | 재작업 요청            |
| 코드 품질 검증   | 린트, 테스트 통과      | 자동 수정 시도         |
| 비용 검증        | 예산 범위 내           | 작업 중단, 승인 요청   |

### 6.5 기술 부채 관리

기술 부채는 빠른 개발을 위해 의도적으로 또는 비의도적으로 발생한 품질 타협입니다.

**부채 유형**:

| 유형              | 예시                               | 이자율    |
| ----------------- | ---------------------------------- | --------- |
| **코드 부채**     | TODO/FIXME, 중복 코드, 복잡한 함수 | 중        |
| **테스트 부채**   | 낮은 커버리지, 깨진 테스트         | 높음      |
| **문서 부채**     | 오래된 문서, 미작성 ADR            | 낮음      |
| **의존성 부채**   | 오래된 패키지, 보안 취약점         | 높음      |
| **아키텍처 부채** | 순환 의존성, God class             | 매우 높음 |

**자동 감지**: 홀론이 작업 완료 시 QualityGate에서 부채 패턴 자동 감지

- TODO/FIXME 주석, 복잡도 초과, 테스트 커버리지 미달, 린트 경고 무시

**이자 개념**: 부채는 시간이 지남에 따라 해결 비용이 증가 (이자 누적)

```

Priority Score = (이자율 × 누적일수) + (영향 파일 수 × 0.5) + (블로킹 태스크 수 × 2)

```

**스프린트별 부채 예산**: 매 사이클마다 전체 용량의 **20%**를 기술 부채 해결에 할당 (고정)

**홀론 책임**:

- 부채 발생 시 즉시 기록 (강제)
- "빠른 해결 vs 부채 생성" 결정은 DecisionLog에 기록

### 6.6 홀론 성과 평가

**평가 주기**: 매 사이클 종료 시 자동 집계

**평가 지표**:

| 지표       | 측정 항목                                      | 가중치 |
| ---------- | ---------------------------------------------- | ------ |
| **생산성** | 완료 태스크 수, 총 스토리포인트                | 40%    |
| **품질**   | 버그 발생률, 코드 리뷰 통과율, 테스트 커버리지 | 35%    |
| **협업**   | 리뷰 참여, 에스컬레이션 적절성, 지식 공유      | 25%    |

**활용**: 태스크 할당 시 역량 기반 매칭, 팀 구성 최적화, 병목 홀론 식별 및 지원

### 6.7 안전장치

#### 서브태스크 재귀 제한

태스크 분할의 무한 재귀를 방지합니다:

| 제한                          | 값                | 초과 시          |
| ----------------------------- | ----------------- | ---------------- |
| 최대 깊이                     | 3단계             | 에스컬레이션     |
| 단일 태스크당 최대 서브태스크 | 10개              | 분할 재검토 요청 |
| 전체 활성 태스크              | Organization 설정 | 신규 생성 대기   |

#### 파일 충돌 방지

여러 홀론이 동시에 같은 파일을 수정하는 것을 방지합니다:

- Task에 `affected_files` 명시 (예상 수정 파일 목록)
- TaskPool에서 파일 경합 감지 시 순차 할당
- 동일 파일 작업 태스크는 동일 홀론에게 우선 배정

#### 롤백/복구 전략

홀론의 작업 결과에 문제 발견 시:

| 상황                   | 대응                              |
| ---------------------- | --------------------------------- |
| 단일 태스크 실패       | Git revert → 태스크 재할당        |
| 연쇄 작업 후 문제 발견 | 의존 태스크 체인 식별 → 순차 롤백 |
| 프로덕션 영향          | P1 Incident 발동 → 인간 개입      |

**원칙**: 모든 태스크는 독립적 커밋 단위로 관리하여 개별 롤백 가능

### 6.8 자율 Goal 생성 (Phase 5+)

> **현재 상태**: Phase 1-4에서는 인간이 Goal을 설정합니다.
> **미래 비전**: Phase 5 이후, 홀론이 자율적으로 Goal을 발견하고 제안할 수 있습니다.

#### 자율 Goal 생성 메커니즘

```

┌─────────────────────────────────────────────────────────────┐
│ 1. 데이터 수집 (자동) │
├─────────────────────────────────────────────────────────────┤
│ - 성과 메트릭: 버그 발생률, 배포 빈도, 사용자 만족도 │
│ - 기술 부채: Priority Score 높은 항목 │
│ - 외부 시그널: 이슈 트래커, 사용자 피드백, 시장 트렌드 │
│ - 팀 상태: Hollon 병목, 반복 작업 패턴 │
└─────────────────────────────────────────────────────────────┘
▼
┌─────────────────────────────────────────────────────────────┐
│ 2. 패턴 인식 및 기회 발견 │
├─────────────────────────────────────────────────────────────┤
│ StrategicAnalyzerService (전담 홀론) │
│ - "보안 취약점 30% 증가" → Goal 후보: "보안 강화" │
│ - "배포 시간 2배 증가" → Goal 후보: "CI/CD 개선" │
│ - "Tech Debt 누적" → Goal 후보: "리팩토링 스프린트" │
└─────────────────────────────────────────────────────────────┘
▼
┌─────────────────────────────────────────────────────────────┐
│ 3. Goal 제안 생성 │
├─────────────────────────────────────────────────────────────┤
│ GoalProposalService │
│ - Objective: "Q2 2024: 시스템 안정성 개선" │
│ - Key Result 1: "평균 응답시간 200ms 이하" │
│ - Key Result 2: "에러율 0.5% 미만" │
│ - Key Result 3: "P1 인시던트 0건" │
│ - 근거: [메트릭 데이터, 이슈 분석, ROI 추정] │
└─────────────────────────────────────────────────────────────┘
▼
┌─────────────────────────────────────────────────────────────┐
│ 4. 인간 승인 (필수) │
├─────────────────────────────────────────────────────────────┤
│ ApprovalRequest 생성 │
│ - request_type: 'goal_proposal' │
│ - 제안 이유, 예상 효과, 리소스 추정 │
│ - 우선순위 제안 (기존 Goal과 비교) │
│ │
│ 인간 리뷰: │
│ ✅ Approve → Goal 생성 → GoalDecomposition 자동 실행 │
│ ❌ Reject → 피드백 저장 → 학습 데이터로 활용 │
│ 📝 Modify → Goal 수정 후 승인 │
└─────────────────────────────────────────────────────────────┘

```

#### 자율 Goal 생성 트리거 조건

| 트리거 유형        | 조건                             | Goal 제안 예시                    |
| ------------------ | -------------------------------- | --------------------------------- |
| **성과 저하**      | 핵심 메트릭 20% 이상 하락        | "성능 최적화"                     |
| **기술 부채 임계** | Priority Score > 100인 부채 5개+ | "Tech Debt 청산 스프린트"         |
| **반복 작업 감지** | 동일 유형 Task 월 10회 이상 발생 | "프로세스 자동화"                 |
| **팀 병목**        | 특정 Hollon에 Task 80% 이상 집중 | "팀 역량 확장" (신규 Hollon 제안) |
| **외부 이벤트**    | 새로운 규제, 경쟁사 동향         | "컴플라이언스 준수", "기능 추격"  |
| **정기 리뷰**      | 분기별 자동 분석                 | "분기 회고 기반 개선 Goal"        |

#### 제약사항 (안전장치)

- **승인 필수**: 자율 생성된 Goal은 반드시 인간 승인 필요
- **생성 빈도 제한**: 월 최대 3개 Goal 제안
- **우선순위 충돌 방지**: 기존 Goal과 리소스 충돌 시 경고
- **투명성**: 제안 근거(데이터, 로직)를 명확히 제시

#### 학습 루프

```

Goal 제안 → 인간 피드백 (승인/거부/수정)
↓
피드백 데이터 축적
↓
StrategicAnalyzerService 개선
↓
더 정확한 Goal 제안 (반복)

```

#### 구현 로드맵

- **Phase 1-4**: 인간이 Goal 설정 ← 현재
- **Phase 5 (미래)**:
  - StrategicAnalyzerService 구현
  - GoalProposalService 구현
  - 메트릭 수집 자동화
- **Phase 6 (장기)**:
  - 학습 루프 고도화
  - 승인 자동화 (신뢰도 기반)
  - 멀티 Goal 최적화

## 8. 협업 및 커뮤니케이션

### 8.1 코드 리뷰 프로세스

**Task-PR 연결**: Linear처럼 Task에 PR(Pull Request)을 연결하여 추적

```

Task #42: "JWT 인증 미들웨어 구현"
│
├── PR #123 (draft) → 작업 중
├── PR #123 (ready_for_review) → 리뷰 요청
├── PR #123 (changes_requested) → 수정 필요
├── PR #123 (approved) → 승인됨
└── PR #123 (merged) → Task 완료

```

**리뷰어 할당 전략**:

| PR 유형        | 리뷰어               | 생성 방식      |
| -------------- | -------------------- | -------------- |
| 일반 코드 변경 | 같은 팀 동료 홀론    | 기존 홀론 활용 |
| 보안 관련      | SecurityReviewer     | 임시 홀론 생성 |
| 아키텍처 변경  | ArchitectureReviewer | 임시 홀론 생성 |
| 성능 민감      | PerformanceReviewer  | 임시 홀론 생성 |

**리뷰 흐름**:

1. 작업 홀론이 PR 생성 (Task 연결)
2. 리뷰어 자동 할당 (팀 동료 또는 전문 Reviewer 홀론)
3. 리뷰 수행 → approve / request_changes
4. 수정 필요 시 작업 홀론이 수정 후 재요청
5. Approve → Merge → Task 완료

> 홀론의 Role은 사람보다 세분화됨: `BackendEngineer` → `TypeScriptReviewer`, `SecurityReviewer` 등

### 8.2 정기 회의 자동화

| 회의 유형       | 트리거         | 결과물                      |
| --------------- | -------------- | --------------------------- |
| 데일리 스탠드업 | 매일 지정 시간 | 진행상황 요약 Document      |
| 스프린트 플래닝 | 사이클 시작    | 태스크 할당 계획            |
| 스프린트 회고   | 사이클 종료    | 회고 Document + 개선 액션   |
| 기술 부채 리뷰  | 주간           | 부채 현황 + 우선순위 재조정 |

### 8.5 불확실성 의사결정

**자율 결정 가능 조건** (모두 충족 시):

1. 되돌릴 수 있는가? ✅
2. 비용 한도 내인가? ✅
3. 외부 영향 없는가? ✅

**Time-boxed 실험 (Spike)**: 불확실성 해소를 위해 시간/비용 제한 내 실험 수행

## 10. 용어 정리

| 용어                  | 정의                                                      |
| --------------------- | --------------------------------------------------------- |
| **Organization**      | 최상위 컨테이너 (회사)                                    |
| **Role**              | 역할 템플릿 (직무 기술서)                                 |
| **Team**              | 홀론들의 그룹 (부서/팀)                                   |
| **Hollon**            | Role 기반 에이전트 인스턴스 (직원)                        |
| **Sub-Hollon**        | 영구 홀론이 생성하는 임시 홀론 (임시 외주)                |
| **Project**           | 목표 달성을 위한 작업 묶음                                |
| **Task**              | 재귀적 작업 단위 (Linear 이슈)                            |
| **Cycle**             | 시간 기반 작업 단위 (스프린트)                            |
| **Brain Provider**    | 홀론의 AI 두뇌 추상화                                     |
| **Single Context**    | 하나의 컨텍스트에서 하나의 태스크 완료 원칙               |
| **Prompt Hierarchy**  | 6계층 프롬프트 합성 구조                                  |
| **Memory (Document)** | Document의 Long-term Memory 역할                          |
| **Escalation**        | 문제 발생 시 단계적 상위 보고                             |
| **Quality Gate**      | 자율 운영의 품질 검증 체크포인트                          |
| **Goal**              | OKR 구조 목표 (Objective/Key Result), 인간 또는 자율 생성 |
| **Decision Log**      | 중요 결정 기록/참조 시스템                                |
| **Fact Check**        | 생성 결과물의 검증                                        |
| **Incident**          | 긴급 대응이 필요한 이슈 (P1-P4)                           |
| **Contract**          | 팀 간 인터페이스/의존성 계약                              |
| **Pivot**             | 전략 방향 변경 (인간 승인 필요)                           |
| **Spike**                | 불확실성 해소를 위한 Time-boxed 실험                               |
| **Pull Request (PR)**    | Task에 연결된 코드 변경 요청                                       |
| **Tech Debt**            | 기술 부채 - 품질 타협으로 인한 미래 비용                           |
| **Code Review**          | PR에 대한 동료/전문 홀론의 검토                                    |
| **READY_FOR_REVIEW**     | 모든 서브태스크 완료 시 부모 태스크가 전환되는 상태 (Phase 3.10)   |
| **IN_REVIEW**            | 부모 홀론이 서브태스크 결과를 LLM으로 검토하는 상태 (Phase 3.10)   |
| **Review Cycle**         | 부모 홀론이 서브태스크 완료 후 결과를 검토하고 결정하는 과정       |
| **Review Decision**      | complete/rework/add_tasks/redirect 4가지 LLM 리뷰 결정 유형 |
| **Temporary Hollon**     | 임시로 생성되어 특정 태스크 완료 후 자동 삭제되는 홀론             |
```
