## Risks & Concerns

### High Risk

**1. Template Content Mismatch**
- **Impact**: Changed prompts → Hollons behave differently → incorrect task execution
- **Probability**: Medium (manual template creation from seed.ts)
- **Mitigation**:
  - Automated content verification script (mandatory in CI)
  - Side-by-side A/B testing in staging
  - E2E tests comparing Hollon outputs before/after
- **Detection**: Integration tests, UAT, production monitoring

**2. Service Integration Bugs**
- **Impact**: Hollon creation failures, system instability
- **Probability**: Low to Medium
- **Mitigation**:
  - Comprehensive integration tests (RoleService ↔ PromptMergeService)
  - Feature flag for gradual rollout
  - Monitor Hollon creation success rates
- **Detection**: E2E tests, error rate alerts

### Medium Risk

**3. Performance Degradation**
- **Impact**: Slower prompt composition (disk I/O, template parsing)
- **Probability**: Medium (if caching not implemented)
- **Mitigation**:
  - Implement caching from day 1 (task #24 in 88d26681-plan)
  - Benchmark P95 latency before production
  - Load testing in staging (1000+ concurrent Hollon creations)
- **Detection**: Latency monitoring, P95 > 100ms alerts

**4. Cache Invalidation Issues**
- **Impact**: Stale templates served after code updates
- **Probability**: Low to Medium
- **Mitigation**:
  - TTL-based cache expiration (5-10 min)
  - Manual cache clear endpoint (`POST /api/cache/clear`)
  - Cache versioning with git commit hash
- **Detection**: Smoke tests, template version mismatch alerts

### Low Risk

**5. Memory Usage Increase**
- **Impact**: Higher memory footprint from template caching
- **Probability**: Low (templates are small, ~2KB each × 14 = ~28KB)
- **Mitigation**:
  - Configure cache size limit (e.g., 100 templates max)
  - Monitor container memory usage
- **Detection**: Container metrics, Kubernetes resource monitoring

**6. Configuration Drift**
- **Impact**: Environment-specific config mismatches
- **Probability**: Low (centralized ConfigService)
- **Mitigation**:
  - Config as code (`.env.example` as SSOT)
  - Environment validation checks on startup
  - Automated config audits
- **Detection**: Startup validation errors
